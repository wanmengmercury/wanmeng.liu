Healthcare Claims Database Agent

The Problem

Healthcare claims databases are notoriously complex: hundreds of tables, arcane medical coding systems (ICD, CPT, NDC), and business logic that varies by claim type, time period, and payer. At our organization, every analytical question — no matter how simple — required an analyst to manually write SQL. Business users waited hours or days for answers. Analysts spent their time on repetitive queries instead of strategic work.

The Insight

The conventional approach to text-to-SQL treats it as a translation problem: take natural language, output SQL. But claims databases don't work that way. The same question — "show me inpatient costs" — requires completely different SQL depending on which table you query, which codes you use, and which business rules apply.

I reframed the problem: text-to-SQL isn't translation, it's decision-making. Writing a correct query requires dozens of sequential choices — which table, which join, which code system, which calculation method. Each choice is small enough to control and debug independently. Rather than asking an LLM to solve everything at once, I built a system that makes these decisions explicit, traceable, and correctable.

This meant the system could explain why it wrote the SQL it did — which matters when business users are putting outputs directly into board presentations.

Technical Approach

The core architecture decomposes each query into discrete decision points. When a user asks about hospital costs, the system first determines claim type (inpatient vs. outpatient), then selects the appropriate table, then identifies the right medical codes, then applies the correct calculation logic. An orchestration layer compiles these decisions into coherent SQL, maintaining clear traceability — when something breaks, I know exactly which decision was responsible.

I built comprehensive tooling for the domain complexity that makes healthcare claims uniquely difficult. This includes lookup tools for ICD, CPT, and NDC codes; logic handlers for claim type selection and table joins; and retrieval from a vector database of validated SQL examples. The system uses prompts and few-shot examples to ensure consistent application of business rules across queries. A self-correction mechanism attempts up to five iterations to fix execution errors before returning results.

To close the feedback loop, I built an evaluation dashboard tracking user engagement and AI-generated gap analysis, plus an automated SQL judge that replaced manual review of unit tests. I designed and implemented the entire UI myself — custom CSS and HTML — since there was no UI developer on the team. The dashboard gave me visibility into both usage patterns and quality gaps, enabling targeted improvements that drove accuracy from early iterations to production-ready performance.
