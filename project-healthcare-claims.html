Healthcare Claims Query Agent

Text-to-SQL on a 200+ table insurance claims database. ~85% accuracy. Outputs used directly in executive reports.

The Problem

Every analytical question against insurance claims database required a human analyst writing SQL — hours per query, days for complex ones. The database had hundreds of tables, four medical coding systems (ICD-10, CPT, DRG, GPI), and business logic that changes depending on claim type, table, and context.

The Insight

Text-to-SQL is usually treated as translation: natural language in, SQL out. That fails on claims data because the same question requires different SQL depending on which table you hit, which code system you filter on, and which business rules apply. The reframe: SQL generation is a sequence of constrained decisions, not a monolithic translation. Each decision — table selection, join logic, filter strategy, calculation method — has its own failure modes and its own context requirements. Decompose them, and you can debug, evaluate, and improve each one independently.

The architecture below reflects this decomposition — each phase isolates a class of decisions with its own context and failure modes.

Architecture

Technical Approach

Why decompose? Left unconstrained, the LLM defaults to the largest available table and the most granular filter — ICD-10 codes for inpatient queries when DRG is the correct abstraction, or filtering on all 7 GPI levels for pharmacy when only GPI-2 and GPI-4 matter. Each SQL decision node receives only the context it needs to make its specific choice correctly. Context retrieval as a parallel system. The question is embedded once, then used across six vector searches simultaneously — drug codes, procedures, diagnoses, DRGs, few-shot examples, and a 1000+ column data dictionary via hybrid vector/keyword search. This front-loads the domain knowledge the decision nodes need without overloading any single prompt. Input quality matters. Users type abbreviations, vague follow-ups, incomplete references. A reformulation step expands and resolves these before anything downstream touches them — reducing the cognitive load on users while giving the system cleaner requirements to work with. Self-correction on execution. Failed queries get rerouted with the error code as context. Usually fixed in one retry. Capped at five.

Impact

  • ~85% accuracy on complex healthcare claims queries
  • 40% productivity increase among users who adopted the system
  • Outputs used in executive slide decks and reports
  • Sole engineer: architecture through frontend
