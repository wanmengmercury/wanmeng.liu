Multi-Agent Analytics Platform for Provider Network Optimization

I designed and deployed a production AI system after an off-the-shelf solution failed at 25% accuracy. Four specialized agent paths, 10 GEPA-evolved LLMs, one unified interface.

Python Claude Sonnet AWS Bedrock DSPy Snowflake Streamlit

The Problem

The starting point was a commercial text-to-SQL solution that hit ~25% accuracy on provider performance queries. Unusable. The domain logic was too proprietary: calculation rules that change year-over-year, distinctions like servicing vs. billing provider, member-level vs. claim-level counting. No pre-trained LLM model knows this. Off-the-shelf solution underperformed, so I designed and built a custom system from scratch.

What I Designed

I architected four agent paths behind a single conversational interface, each solving a different class of question:

I broke the text-to-SQL problem into 10 independent decision-point LLMs: one for join logic, one for WHERE clauses, one for specialty filters, one for count semantics, and so on — rather than forcing a single prompt to handle everything. This decomposition was deliberate: when accuracy drops, I can trace the failure to a specific decision point and fix it in isolation. I built a self-correcting layer on top that validates and rewrites assembled SQL before execution.

For document retrieval, I designed a multi-knowledgebase architecture with ~10 domain-specific KBs rather than one monolithic index. I separated KBs by document type because I discovered that methodology docs from consecutive years look nearly identical to embedding models but must rank equally for year-over-year comparison. Mixing them into one index buried older versions. Partitioning solved this.

I also built a scenario modeling path. It is a form-guided simulation backed by a linear solver that lets users test how per-member-per-year cost shifts under different provider/network configurations.

How I Scaled Prompt Engineering with GEPA

10 LLMs means 10 prompts, each encoding proprietary logic that exists nowhere in any model's pretraining. I couldn't hand-tune all of them. So I built a self-improving training loop using GEPA (Genetic-Pareto optimization via DSPy).

I started with 20 hand-written positive and negative examples per LLM. GEPA evolves candidate prompts through reflective mutation: it reads execution traces to diagnose why a prompt failed, then proposes targeted fixes which are not random perturbation. I scored candidates via cosine similarity against ground truth, then human-reviewed rollout outputs to catch domain-incorrect rules GEPA proposed. Verified outputs became new training data; wrong-but-close outputs I corrected manually and fed back in. Each cycle gave me better prompts and a larger dataset.

This workflow is what let me optimize 10 prompts simultaneously and push accuracy from 25% to 70% in one week — instead of spending that week hand-picking words and examples for a single prompt.
