Provider Performance Agent

Outperformed Snowflake Intelligence by decomposing text-to-SQL into traceable decision points

<20s response time
6-9× faster evaluation
Full-stack ownership
Text-to-SQL Multi-Agent Snowflake Streamlit LangChain GEPA Optimization

The Problem

Healthcare organizations sit on rich provider performance data—cost efficiency, care appropriateness, quality metrics—but accessing it requires analysts who understand both the business logic and the underlying data model. Business users wait days for answers. Analysts spend hours on repetitive queries instead of deep analysis.

The obvious solution is natural language querying, but enterprise AI platforms consistently fail on complex proprietary datasets. When I evaluated Snowflake Intelligence on the provider performance data, the accuracy gaps were immediately disqualifying.

The Insight

Most text-to-SQL systems treat query generation as a single problem: user question in, SQL out. When the query fails, you're left debugging a black box. I realized that writing a correct query isn't one decision—it's a sequence of discrete decisions, each with its own failure modes:

  • Which tables contain the relevant data?
  • How should they be joined?
  • What filters apply to this question?
  • Which metrics need calculation, and what's the correct formula?
  • How should results be aggregated and ordered?

By decomposing the problem this way, I could build an orchestration layer that makes each decision explicit and traceable. When a query returns wrong results, I know exactly which decision failed—was it table selection? Join logic? A calculation rule applied incorrectly? This traceability transformed debugging from guesswork into systematic diagnosis.

The second insight came during a compliance-driven platform migration. The organization was stuck in an unresolved argument: AWS services offered better AI capabilities, but Snowflake satisfied compliance requirements. I recognized this as a false binary. By mapping the architectural dependencies carefully, I found a migration path that preserved the text-to-SQL accuracy (the core value) while accepting some degradation in document retrieval performance (a secondary feature). This tradeoff broke the deadlock—not by finding a perfect solution, but by making the costs and benefits explicit enough that stakeholders could align.

Technical Approach

Architecture. I built a master routing agent that receives user questions and directs them to specialized components: a SQL-generation agent for data queries, a document-retrieval agent for policy and methodology questions, or a decline response when the question falls outside scope. This consolidation meant users had one interface for all provider performance needs, rather than context-switching between tools.

Decision decomposition. The SQL agent doesn't generate queries in one shot. It works through a structured sequence: identifying relevant tables, determining join paths, selecting appropriate filters, applying calculation rules, and composing the final query. Each step produces an intermediate artifact I can inspect. An orchestration layer compiles these decisions into executable SQL while maintaining full traceability.

Self-correction loop. Queries that fail execution enter a correction cycle—up to five iterations of error analysis and query revision before returning results or acknowledging failure. This significantly improved success rates on complex questions without sacrificing transparency about what the system can and cannot do.

Dual-audience design. Business users need answers and understand filters but don't write SQL. Analytical users need the SQL itself to verify logic or extend the analysis. I designed outputs to serve both: natural language summaries with the underlying query available on demand. This wasn't just a UI decision—it shaped how I structured the agent's reasoning to remain interpretable at every step.

Full-stack ownership. With no UI developer on the team, I built the entire frontend—starting with custom CSS/HTML injections in Streamlit. The application handles conversation history, result visualization, feedback capture, and usage analytics.

Evaluation infrastructure. I built an automated SQL judge that evaluates unit test results against expected outputs, reducing evaluation time from 2-3 hours of manual review to 20 minutes of human oversight. Faster feedback loops meant faster iteration and higher final accuracy.

Impact

  • Dramatically outperformed Snowflake Intelligence on provider performance data—accuracy differences led to strategic decision to build on custom architecture rather than enterprise platform
  • Resolved organizational deadlock on environment requirements by executing AWS-to-Snowflake migration that satisfied compliance while preserving core functionality
  • Enabled self-service querying for both business users (who previously waited for analyst support) and analytical users (who now get query logic they can verify and extend)
  • 6-9× speedup in evaluation cycles through automated SQL judge, enabling rapid iteration during development
  • Sub-20-second response time on queries that previously took analysts minutes to hours
